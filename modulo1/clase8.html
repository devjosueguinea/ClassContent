Clase 8: Elaboración del Diccionario de Datos

El Diccionario de Datos es un insumo crucial en el análisis y diseño de sistemas informáticos. Funciona como un catálogo centralizado que recopila los metadatos de las entidades de base de datos, definiendo con precisión la estructura, formato, nulabilidad y reglas de negocio aplicadas a la información del sistema.

¿Qué es un Diccionario de Datos y por qué es Importante?

En proyectos de software empresarial, diferentes equipos (diseñadores de bases de datos, desarrolladores frontend, ingenieros backend y analistas de negocio) necesitan comunicarse sin malentendidos semánticos. El diccionario de datos actúa como el punto único de verdad (Single Source of Truth).

📂 Control de Metadatos

No almacena los datos reales del cliente (como "Juan Pérez"), sino los metadatos (definición de la columna: `nombre`, tipo `VARCHAR2(100)`, obligatorio).

🤝 Consistencia de Equipo

Evita inconsistencias donde un programador nombre un campo como `mail` en la base de datos, otro como `correo` en el backend, y otro como `emailAddress` en las vistas.

⚙️ Reglas de Negocio

Documenta de forma clara restricciones de dominio (constraints) como llaves primarias, foráneas, valores únicos, nulabilidad y valores por defecto.

Componentes Esenciales de un Diccionario de Datos

Al diseñar el plano de almacenamiento del sistema, cada campo de una entidad debe estar completamente parametrizado. Despliega cada punto para conocer qué información debe contener tu plantilla:

✏️ 1. Nombre del Campo físico (Nombre Lógico vs. Físico)
Es el identificador técnico del atributo en la tabla (ej. `cliente_id`, `correo_electronico`). Debe seguir una convención de nomenclatura estándar (como *snake_case* o *camelCase*) y ser descriptivo.
🔢 2. Tipo de Dato y Longitud
Define la naturaleza de la información almacenada (ej. `VARCHAR2`, `NUMBER`, `DATE`, `BOOLEAN`) y la capacidad máxima de almacenamiento permitida para ese atributo (ej. 100 caracteres, 10 dígitos decimales).
⚪ 3. Nulabilidad (NULL / NOT NULL)
Especifica si es obligatorio que el campo contenga un valor al insertar un registro (`NOT NULL`) o si puede dejarse en blanco de forma opcional (`NULL`).
🔑 4. Llaves (Keys: PK / FK)
Identifica si el campo actúa como la clave primaria de la tabla (**Primary Key - PK**), la cual garantiza registros únicos no nulos, o si es una referencia a otra tabla (**Foreign Key - FK**), asegurando la integridad referencial.
🛠️ 5. Valores por Defecto y Restricciones (Constraints)
Documenta valores asignados automáticamente por el sistema si el usuario no introduce ninguno (ej. `SYSDATE` para fechas de registro) y validaciones adicionales de dominio mediante restricciones de tipo **CHECK** (ej. `edad >= 18`).

Plantilla Interactiva: Entidad `Clientes`

A continuación se presenta el estándar recomendado para estructurar el diccionario de datos físico para una tabla típica de almacenamiento en bases de datos:

Nombre de Campo Tipo de Dato Longitud Nulabilidad Llave Valor por Defecto Descripción y Restricciones (Constraints)
cliente_id NUMBER 10 NOT NULL PK Ninguno Identificador único autoincremental asignado a cada cliente.
nombre_completo VARCHAR2 100 NOT NULL - Ninguno Nombres y apellidos completos legales del cliente.
correo_electronico VARCHAR2 150 NOT NULL - Ninguno Dirección de correo. Debe ser única globalmente (Constraint: UNIQUE).
fecha_registro DATE - NOT NULL - SYSDATE Fecha y hora automática en la que el cliente fue creado en la base de datos.
estado_cuenta VARCHAR2 15 NOT NULL - 'ACTIVO' Estado operativo. Restricción (CHECK): solo permite 'ACTIVO', 'INACTIVO', 'SUSPENDIDO'.
telefono_contacto VARCHAR2 20 NULL - Ninguno Número de contacto telefónico del cliente (opcional).

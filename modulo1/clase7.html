Clase 7: Relación de Agregación y Relación de Composición

En el modelado de clases UML y la Programación Orientada a Objetos, las relaciones de Agregación y Composición definen cómo los objetos se relacionan estructuralmente en términos de jerarquías "todo-parte". Aunque ambas describen contención, su acoplamiento y control de ciclo de vida son sustancialmente diferentes.

Parte 1: Relación de Agregación (Asociación Débil)

La agregación describe una relación unidireccional de tipo "todo-parte" donde la existencia del objeto "parte" es independiente de la del objeto "todo". Esto significa que si el objeto contenedor es destruido, los objetos contenidos continúan existiendo de forma autónoma.

👥 Ciclo de Vida Independiente

El contenedor (todo) no gestiona de forma directa la creación o destrucción de la parte. Las partes son instanciadas externamente y asociadas al contenedor dinámicamente.

💼 Ejemplo del Mundo Real

Una Universidad y sus Profesores. Si la universidad cierra sus puertas, los profesores no dejan de existir en el mundo; simplemente se desvinculan de esa institución y pueden agregarse a otra.

💎 Notación UML

Se representa utilizando una línea sólida que finaliza con un rombo vacío (sin rellenar) en el extremo conectado a la clase contenedora (el "todo").

Ejemplo de Código en Java (Agregación)

Observa cómo el profesor es inyectado a la Universidad en su constructor, existiendo fuera de ella:

// Clase Parte
public class Profesor {
    private String nombre;

    public Profesor(String nombre) {
        this.nombre = nombre;
    }
}

// Clase Contenedora (Todo)
public class Universidad {
    private String nombre;
    private List<Profesor> profesores; // Agregación

    // Los profesores se crean afuera y se inyectan a la Universidad
    public Universidad(String nombre, List<Profesor> profesores) {
        this.nombre = nombre;
        this.profesores = profesores;
    }
}
Parte 2: Relación de Composición (Asociación Fuerte)

La composición es una forma más restrictiva y fuerte de agregación donde el objeto "parte" no puede existir independientemente del objeto "todo". El ciclo de vida de la parte es administrado enteramente por el contenedor, coincidiendo con la creación y destrucción de este.

🔒 Ciclo de Vida Coincidente

Cuando el objeto contenedor es destruido, el recolector de basura (o destructor) elimina automáticamente todas sus partes constitutivas. Las partes no se pueden transferir a otro contenedor.

🖥️ Ejemplo del Mundo Real

Una Computadora y su CPU (procesador). La CPU se diseña e instala para ser parte integral de la computadora en su fabricación. Si destruyes físicamente la computadora, su procesador interno también se pierde con ella.

🖤 Notación UML

Se representa con una línea sólida que finaliza con un rombo relleno (color negro o sólido) en el extremo conectado a la clase contenedora (el "todo").

Ejemplo de Código en Java (Composición)

El objeto CPU se crea en el constructor de la Computadora, atando su ciclo de vida al contenedor:

// Clase Parte
public class CPU {
    private String modelo;

    public CPU(String modelo) {
        this.modelo = modelo;
    }
}

// Clase Contenedora (Todo)
public class Computadora {
    private String marca;
    private CPU cpu; // Composición

    // El objeto CPU es instanciado internamente en el constructor
    public Computadora(String marca, String modeloCPU) {
        this.marca = marca;
        this.cpu = new CPU(modeloCPU);
    }
}

Comparación de Agregación y Composición

Esta tabla resume los factores diferenciales clave entre ambas relaciones:

Característica Agregación (Aggregation) Composición (Composition)
Ciclo de Vida Independiente de la clase contenedora. Coincidente. Se destruyen juntos.
Dependencia Débil. La parte puede vivir sin el todo. Fuerte. La parte no tiene sentido sin el todo.
Gráfico UML Línea sólida con rombo vacío ♢. Línea sólida con rombo relleno ♦.
Implementación en Java Los objetos se pasan como parámetros. Los objetos se crean usando new dentro del constructor.
Multiplicidad de Relación Las partes pueden pertenecer a múltiples agregadores. Las partes pertenecen a un único contenedor a la vez.

El Acceso Seguro: Conexión a Oracle y Variables de Entorno

Cuando trabajamos en entornos profesionales, **nunca** debemos exponer credenciales (usuario, contraseña, URLs) en código duro dentro de nuestro archivo de configuración application.properties o subirlos a repositorios como GitHub. Para esto utilizamos variables de entorno y un archivo local .env.

🛡️ Importancia del archivo .env: El archivo .env reside de forma local y se añade a .gitignore. Permite definir variables locales que la aplicación leerá en tiempo de ejecución. Si un intruso accede a tu código fuente, no podrá comprometer la base de datos de producción porque las claves se inyectan dinámicamente en el entorno de ejecución.

Define las credenciales en la raíz del proyecto para evitar subirlas a Git:

DB_URL=jdbc:oracle:thin:@localhost:1521:xe
DB_USER=mi_usuario_oracle
DB_PASSWORD=mi_clave_secreta

Añade la librería dotenv-java en tu archivo de dependencias (por ejemplo, Maven pom.xml):

<dependency>
    <groupId>io.github.cdimascio</groupId>
    <artifactId>dotenv-java</artifactId>
    <version>3.0.0</version>
</dependency>

Lee las variables dinámicamente usando la sintaxis de Spring Boot:

spring.datasource.url=${DB_URL}
spring.datasource.username=${DB_USER}
spring.datasource.password=${DB_PASSWORD}
spring.datasource.driver-class-name=oracle.jdbc.OracleDriver
spring.jpa.database-platform=org.hibernate.dialect.OracleDialect

Carga el archivo .env al sistema antes de arrancar Spring Boot:

import io.github.cdimascio.dotenv.Dotenv;

@SpringBootApplication
public class Application {
    public static void main(String[] args) {
        // Cargar variables del .env a System Properties
        Dotenv dotenv = Dotenv.configure().ignoreIfMissing().load();
        dotenv.entries().forEach(entry -> 
            System.setProperty(entry.getKey(), entry.getValue())
        );
        SpringApplication.run(Application.class, args);
    }
}

El Modelo y la Persistencia Automática

Spring Data JPA nos permite interactuar con la base de datos Oracle a través de objetos de Java (Entities) y una interfaz de persistencia preconstruida llamada JpaRepository.

La Entidad JPA (@Entity)

Mapea una clase Java directamente a una tabla física en Oracle.

@Entity
@Table(name = "ESTUDIANTES")
public class Estudiante {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;
    
    @Column(name = "NOMBRE", nullable = false)
    private String nombre;
    
    @Column(name = "CORREO", unique = true)
    private String correo;

    // Getters y Setters
}

La Interfaz Repository

Hereda de JpaRepository para obtener operaciones CRUD de inmediato.

@Repository
public interface EstudianteRepository 
    extends JpaRepository<Estudiante, Long> {
    
    // Método personalizado automático (Query Method)
    Optional<Estudiante> findByCorreo(String correo);
}
💡 Ventaja: No necesitas escribir INSERT INTO, UPDATE, DELETE ni SELECT. Spring Data JPA genera el SQL por ti automáticamente tras bambalinas.

Capa de Servicio: La Lógica de Operaciones CRUD

La capa de servicio coordina el repositorio para ejecutar operaciones CRUD seguras, aplicando lógica de negocio y excepciones si un recurso no existe.

Implementación en EstudianteService

Utilizamos inyección de dependencias para integrar el repositorio y anotamos la clase con @Service.

@Service
public class EstudianteService {

    @Autowired
    private EstudianteRepository repository;

    // CREATE (Crear)
    public Estudiante guardar(Estudiante estudiante) {
        return repository.save(estudiante);
    }

    // READ ALL (Leer todos)
    public List<Estudiante> obtenerTodos() {
        return repository.findAll();
    }

    // READ BY ID (Leer por ID)
    public Estudiante obtenerPorId(Long id) {
        return repository.findById(id)
            .orElseThrow(() -> new RuntimeException("Estudiante no encontrado"));
    }

    // UPDATE (Actualizar)
    public Estudiante actualizar(Long id, Estudiante detalles) {
        Estudiante estudiante = obtenerPorId(id);
        estudiante.setNombre(detalles.getNombre());
        estudiante.setCorreo(detalles.getCorreo());
        return repository.save(estudiante);
    }

    // DELETE (Eliminar)
    public void eliminar(Long id) {
        Estudiante estudiante = obtenerPorId(id);
        repository.delete(estudiante);
    }
}

Capa de Presentación: Exposición de Endpoints HTTP

El controlador recibe las peticiones externas, invoca a nuestro servicio de negocio y despacha las respuestas en JSON acompañadas del respectivo código de estado HTTP (200 OK, 201 Created, 204 No Content).

Controlador API REST: EstudianteController

@RestController
@RequestMapping("/api/estudiantes")
public class EstudianteController {

    @Autowired
    private EstudianteService service;

    @GetMapping
    public ResponseEntity<List<Estudiante>> listar() {
        return ResponseEntity.ok(service.obtenerTodos());
    }

    @GetMapping("/{id}")
    public ResponseEntity<Estudiante> obtener(@PathVariable Long id) {
        return ResponseEntity.ok(service.obtenerPorId(id));
    }

    @PostMapping
    public ResponseEntity<Estudiante> crear(@RequestBody Estudiante estudiante) {
        return new ResponseEntity<>(service.guardar(estudiante), HttpStatus.CREATED);
    }

    @PutMapping("/{id}")
    public ResponseEntity<Estudiante> actualizar(@PathVariable Long id, @RequestBody Estudiante estudiante) {
        return ResponseEntity.ok(service.actualizar(id, estudiante));
    }

    @DeleteMapping("/{id}")
    public ResponseEntity<Void> eliminar(@PathVariable Long id) {
        service.eliminar(id);
        return ResponseEntity.noContent().build();
    }
}
